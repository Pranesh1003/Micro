==============================================================

How to read this file
---------------------
Every entry below starts with an exact source file and source-line number(s).
Where Java uses a multi-line statement or a block of purely structural lines
(braces, imports, getters/setters), the adjacent lines are explained together;
the explanation still accounts for every meaningful code line. Blank lines only
separate code sections and have no runtime effect. `api-service` in the request
is the `api-gateway` module in this repository.

BOOKING END-TO-END CODE PATH
  1. Browser asks gateway `/flights/search`; gateway routes it to Booking.
  2. BookingServiceImpl calls Feign `FlightClient.getAllFlights` by service name.
  3. User submits passenger details; Booking creates/validates a Booking draft.
  4. Seat page retrieves `/airline/api/seats` and shows AVAILABLE/BOOKED seats.
  5. Confirmation transaction changes one seat to BOOKED, builds a mock Payment
     record and returns its generated PNR-like booking identifier.
  6. Cancellation changes status and attempts to release that selected seat.

Common JWT implementation note
  Auth, Flight and Booking each have separate `JwtUtils` and `JwtRequestFilter`
  source files. Their line-by-line logic is intentionally almost identical: a
  token signed by Auth is accepted by the other services only because they use
  the same secret/default configuration and HS256 algorithm.

Important code-level limitations identified while explaining
  * Gateway central routes use hard-coded localhost ports instead of `lb://`.
  * Passwords are compared/stored as plain text in AuthController/User.
  * Booking endpoints are mostly permitted without JWT in Booking SecurityConfig.
  * MockPaymentService is not a payment gateway; it only generates a random PNR.
  * Flight and Booking each access seat inventory representations while sharing
    one MySQL schema. A production microservice design should give one service
    ownership and communicate seat changes by API/event.
